# P의 여행 (P of Travel)

> **계획은 없어도 괜찮아.**
> 실시간 관광 빅데이터가 오늘 떠날 소도시를 골라주는 즉흥 여행 서비스

*2025 관광데이터 활용 공모전 · 생성형 AI 활용 관광 프롬프톤 부문*

| 타겟 | 핵심 미션 | TourAPI 활용 | 개발 기간 |
|---|---|---|---|
| MBTI P 여행자 | 소도시 관광 분산 | 9종 | 5일 |

---

## 01. 문제 정의

### 가고 싶은데, 어디 갈지를 모른다

일요일 아침, 30대 1인 가구 A씨. 잠에서 깼는데 날씨가 너무 좋다. 어디든 가고 싶다. 그런데 어디로 가야 할지 모른다.

| A씨가 하는 행동 | 결과 |
|---|---|
| 인스타그램을 본다 | 예쁜 곳이 나오지만 거기가 어디인지 모른다 |
| 네이버 지도를 켠다 | 무엇을 검색해야 할지 모른다 |
| 주변에 물어본다 | 다들 비슷하게 갈 곳을 모른다 |
| 결국 그냥 누워있는다 | 좋은 날씨와 의욕이 그대로 사라진다 |

**문제의 본질은 '정보 부족'이 아니라 '결정 장애'다.** 정보는 넘치는데, 지금 내 상황에 맞는 단 하나의 답을 주는 서비스가 없다.

### 동시에 — 관광은 한쪽으로만 쏠린다

서울·부산·제주에 관광객이 과포화되는 동안, 전국의 수많은 소도시는 '거기 뭐가 있는지 몰라서' 외면받는다. 소도시에 콘텐츠가 없어서가 아니다. 데이터로 연결되지 않아서 아무도 모를 뿐이다.

> "P의 여행"은 이 두 문제를 동시에 푼다. 즉흥 여행자의 결정 장애를 해결하면서, 그 결정을 데이터 기반으로 소도시에 연결한다.

## 02. 서비스 컨셉

### 한 줄 정의

> **날씨 · 이동수단 · 시간 · 테마, 4가지만 고르면 실시간 관광 빅데이터가 '지금 떠나기 가장 좋은 소도시'를 골라 사진과 함께 코스로 보여주는 즉흥 여행 서비스**

### 왜 'P'인가

| | J의 여행 | P의 여행 (우리 서비스) |
|---|---|---|
| 계획 시점 | 몇 주 전 예약·일정표 | 오늘 아침에 결정 |
| 정보 탐색 | 리뷰 정독, 블로그 비교 | 안 함. 바로 답을 원함 |
| 필요한 것 | 완벽한 계획 | '그냥 가도 된다'는 확신 |
| 서비스가 주는 것 | — | 데이터 근거 + 비주얼 + 즉시 출발 |

P는 설득이 아니라 자극이 필요하다. 긴 설명 대신, 좋은 사진 한 장과 '지금 가세요'라는 한 줄이 P를 움직인다.

## 03. 사용자 입력 — 무엇을 묻는가

입력을 두 종류로 분리한다. 매번 바뀌는 '오늘의 조건'과, 처음 한 번만 받는 '기본 설정'이다.

### A. 온보딩 — 처음 한 번, 단 하나

| 질문 | 용도 | 비고 |
|---|---|---|
| 어디서 출발하세요? | 출발지 기본값 설정 | 매번 GPS 허용 안 해도 됨. GPS 자동 감지도 가능 |

P에게 긴 온보딩은 이탈의 지름길이다. 취향은 묻지 않는다. 쓰면서 학습한다.

### B. 오늘의 조건 — 쓸 때마다, 4개 탭

| 조건 | 선택지 | AI 추천에서의 역할 |
|---|---|---|
| 언제 | 오늘 / 내일 / 이번 주말 / 날짜 직접 선택 | 혼잡도 예측 기준일 + 행사 매칭 |
| 이동수단 | 자차 / 대중교통(KTX·버스) | 도달 가능 반경 + 소요시간 계산 |
| 얼마나 | 반나절 / 하루 / 1박 이상 | 반경 크기 + 코스 분량 결정 |
| 테마 | 감성 / 힐링 / 액티비티 / 맛집 / 문화 | 장소 카테고리 + 혼잡도 선호 역추론 |
| (선택) 누구랑 | 혼자 / 친구 / 연인 / 가족 | 관광 다양성 API 연령 데이터 매칭 |

**날씨는 묻지 않는다.** 시스템이 출발지·목적지 날씨를 자동 감지하여 추천에 반영한다.

### C. 취향은 '입력'이 아니라 '축적'된다

취향을 글로 표현하는 것은 어렵다. 그래서 직접 묻지 않는다.

- 오늘의 '테마' 선택이 곧 오늘의 취향 표현이다. (감성→한산 선호, 액티비티→활기 선호 등 AI가 맥락 해석)
- 여행 다녀온 다음날 푸시 1회: "○○ 어땠어요?" → **[좋았어] [별로]** 단 두 개 버튼
- 3회 이상 누적되면 패턴을 학습해 자동 반영 (Phase 2)

## 04. 추천 엔진 — 어떻게 소도시를 고르는가

**핵심 설계 원칙:** '점수 가중치'를 임의로 정하지 않는다. 데이터 수집·반경 계산은 로직이 처리하고, 최종 판단과 설명은 생성형 AI(Claude)가 맥락으로 수행한다. 역할을 명확히 분리하는 것이 이 서비스의 기술적 핵심이다.

### 전체 파이프라인 (6단계)

| 단계 | 담당 | 내용 |
|---|---|---|
| ① 반경 계산 | `로직` 카카오맵 API | 현재 위치/출발지 + 이동수단 + 시간 → 실시간 교통 반영 도달 가능 시군구 목록 추출 |
| ② 소도시 풀 생성 | `로직` 행정구역 필터 | 추출된 시군구 중 8대 대도시 제외(서울·부산·인천·대구·광주·대전·울산·세종) → 소도시 후보 풀 |
| ③ 데이터 수집 | `API 병렬 호출` | 후보 소도시 각각에 대해 수요강도·다양성·자원수요·날씨·혼잡도 수치 동시 수집 |
| ④ 최종 판단 | `생성형 AI` Claude | 수집 데이터 + 사용자 조건(테마·동행)을 컨텍스트로 주입 → 가장 적합한 소도시 1곳 선정 + 선정 이유 한 줄 생성 |
| ⑤ 코스 생성 | `API + AI` | 확정 소도시 내 장소 추출 → 시간순 코스(오전·점심·오후) 구성 |
| ⑥ 비주얼·경로 | `API` | 장소별 사진 매칭 + 소요시간/길찾기 경로 생성 |

### 왜 반경 계산을 AI에게 맡기지 않는가

"서울에서 2시간 내 소도시"를 Claude에게 직접 물으면 학습된 지식으로 답해 실시간 교통이 반영되지 않고 환각 위험이 있다. 반경·후보 추출은 반드시 카카오맵 API 로직으로 처리하고, AI는 '후보 중 무엇이 오늘 최적인가'라는 판단에만 집중시킨다.

### 혼잡도는 '낮을수록 좋다'가 아니다

혼잡도에는 정답이 없다. 조용한 곳을 원하는 사람도, 활기를 원하는 사람도 있다. 따라서 혼잡도를 점수에 직접 더하지 않는다. 대신 테마에서 선호를 역추론해 Claude가 해석한다.

| 선택 테마 | 혼잡도 선호 (AI 해석) | 근거 데이터 |
|---|---|---|
| 감성 | 한산한 곳 선호 | 사진·여유 중시 → 집중률 낮은 곳 가중 |
| 힐링 | 한산한 곳 강하게 선호 | 자연·휴식 → 집중률 낮을수록 우선 |
| 액티비티 | 적당한 활성화 필요 | 시설 운영 위해 일정 수요 필요 |
| 맛집 | 혼잡도 무관 | 소비강도 높은 곳이 더 중요 |
| 문화 | 혼잡도 무관 | 문화자원 수요 높은 곳 우선 |

## 05. TourAPI 활용 상세 (9종)

각 API가 파이프라인의 어느 단계에서, 어떤 데이터를, 왜 쓰는지를 명시한다.

| API명 | 수집 데이터 | 서비스 내 역할 |
|---|---|---|
| 국문 관광정보 서비스 (위치기반) | 반경 내 관광지·음식점·문화시설 | 코스 장소 후보 풀 + 이미지 정보 |
| 관광지 집중률 방문자 추이 예측 | 향후 5주 날짜별 방문자 비율(머신러닝) | 이번 주말·미래 날짜 혼잡도 예측 |
| 지역별 관광 수요 강도 | 체류강도(숙박·타권역 방문), 소비강도 | 오래 머물 만한 곳 / 먹고 쓸 곳 판단 |
| 지역별 관광 다양성 | 연령별 방문·소비, 국적별 방문자 | 동행 유형(혼자·가족) 연령 매칭 |
| 지역별 관광 자원 수요 | SNS 언급량, 내비 목적지 검색량 | 지금 막 뜨기 시작한 소도시 감지 |
| 관광지별 연관 관광지 정보 | 중심관광지와 연결성 높은 장소 | 코스 동선 + 1박 시 인접 소도시 연결 |
| 관광공모전 수상작 사진 | 고퀄리티 촬영 사진 | 결과 카드 메인 비주얼 (핫플 위주) |
| 관광사진 정보 | 촬영장소·촬영일 포함 사진 | 소도시·계절감 사진 보완 (커버리지) |
| 행사·축제 정보 | 기간·위치 기반 행사 일정 | '오늘/이번 주 거기서 뭔가 열림' 매칭 |

### 외부 무료 API (보조)

| API | 제공처 | 역할 |
|---|---|---|
| 기상청 단기예보 | 공공데이터포털 | 출발지·목적지 날씨/강수 자동 감지 |
| 카카오맵 (길찾기·대중교통) | 카카오 OpenAPI | 반경 계산·소요시간·경로 |
| 코레일 열차정보 | 공공데이터포털 | KTX 시간표·요금·잔여석 |
| Claude API | Anthropic | 최종 판단·설명·코스 내러티브 생성 |

### 비주얼 커버리지 문제와 해법

공모전 수상작 사진은 퀄리티가 높지만 핫플에 편중돼, 소도시 추천 서비스와 충돌한다. 따라서 세 단계로 사진을 확보한다.

- **1순위:** 공모전 수상작 사진 API — 가장 고퀄리티
- **2순위:** 관광사진 정보 API + TourAPI 이미지 정보 — 소도시 커버리지 보완
- **사진이 없는 소도시:** '아직 아무도 담지 않은 곳'으로 카피 전환 — P에게는 미개척지가 오히려 매력

## 06. 결과 화면 — 사진으로 끌고, 코스로 잡는다

P를 움직이는 것은 텍스트가 아니라 비주얼이다. 결과는 풀스크린 사진 슬라이드로 코스를 탐색하는 형태로 구성한다.

### 화면 구성

| 요소 | 내용 |
|---|---|
| 메인 비주얼 | 추천 소도시 풀스크린 사진 (오늘 날씨·계절 반영 컷) |
| 타이틀 + 한 줄 | 소도시명 + Claude 생성 추천 이유 ("오늘 이 날씨엔 여기예요") |
| 교통 정보 | 자차: 지금 출발 시 소요시간 / 대중교통: KTX·버스 시간 |
| 코스 (스와이프) | 오전·점심·오후 장소를 사진 넘기며 탐색, 각 장소 한 줄 설명 |
| 혼잡도 배지 | 한산 / 보통 / 붐빔 (집중률 예측 기반) |
| CTA | [지금 출발할게요] → 카카오맵 코스 길찾기 연동 |
| [다시 추천] | P는 마음이 바뀐다. 언제든 새로 뽑을 수 있는 버튼 상시 노출 |

### Claude 설명의 차별점

| 일반 여행 앱 | P의 여행 |
|---|---|
| 담양군. 죽녹원, 메타세쿼이아길 등 | 지금 담양은 외지인 소비가 평소보다 높고, 이번 주말 대나무축제 마지막 날이에요. 혼잡도는 아직 낮음. 지금 가면 딱이에요. |

데이터가 설명의 근거가 되므로 추천에 신뢰감이 생긴다. 단, 사용자에게는 수치를 노출하지 않고 사람의 말로 번역해 한 줄로만 전달한다.

## 07. 주요 시나리오별 처리

**당일치기 (반나절·하루)**
- 소도시 1곳 + 시간순 코스 (오전→점심→오후)

**1박 이상**
- 소도시 1곳 + 이틀 코스, 또는 연관관광지 API로 인접 소도시 연결 (예: 강릉→속초, 공주→부여)

**미래 날짜 / 이번 주말**
- 집중률 예측 API가 향후 5주 날짜별 방문자 비율을 제공 → 미래 혼잡도 예측 가능
- P도 가끔 계획한다. '날짜 직접 선택' 모드로 J 니즈도 흡수

**출발지 ≠ 현재 위치**
- GPS 자동 감지를 기본값으로, 출발지 직접 변경도 지원 (친구 집 등에서 출발하는 경우)

**출발지·목적지 날씨 불일치**
- 서울 맑음 / 강릉 비 같은 경우 대비 → 목적지 날씨가 좋은 소도시를 우선 추천
